## 一、Hermes Agent 是什么

Hermes Agent 是 Nous Research 打造的开源 AI Agent，与 LangGraph、AutoGen 等多 Agent 编排框架**走的是完全不同的路线**：

| 对比维度 | LangGraph / CrewAI 等 | Hermes Agent |
|----------|----------------------|-------------|
| 核心设计 | 多 Agent 编排调度 | **单体 Agent**，自进化 |
| 进化方式 | 手动配置工作流 | **自动从实战中提炼技能** |
| 记忆机制 | 框架提供，用户实现 | **四层内置记忆**，自动管理 |
| 部署模式 | 代码库 / API | **完整产品**（CLI / Gateway / IDE集成 / Cron） |
| 学习门槛 | 框架级别 | 产品级别 |

**一句话定位**：Hermes 不是让你"搭建"Agent，而是直接给你一个**开箱即用、越用越聪明**的 AI 助手产品。

## 四、模块一：AIAgent 核心循环引擎

### 4.1 定位

整个系统的**心脏**，`run_agent.py` 中的 `AIAgent` 类（约 4400 行代码）。所有入口（CLI、Gateway、Cron、ACP、API Server）最终都调用同一个 `AIAgent` 实例。

### 4.2 核心职责

```
AIAgent.run_conversation()
        │
        ├── 生成 task_id（任务追踪 ID）
        ├── 追加用户消息到对话历史
        ├── 构建/复用缓存的系统提示词
        ├── 预检压缩（>50% 上下文时触发）
        ├── 构建 API 消息
        ├── 注入临时提示层（预算警告等）
        ├── 发起可中断的 API 调用
        ├── 解析响应：
        │   ├── 有 tool_calls → 执行工具 → 结果追加 → 返回构建API消息
        │   └── 文本响应 → 持久化会话 → 刷新内存 → 返回结果
        └── 返回最终响应
```

### 4.3 三种 API 模式

Hermes 不绑定单一 LLM 接口，支持三种 API 模式，根据 Provider 自动选择：

| API 模式 | 用途 | 客户端类型 |
|----------|------|-----------|
| `chat_completions` | OpenAI 兼容端点（大多数国内厂商） | `openai.OpenAI` |
| `codex_responses` | OpenAI Codex / Responses API | `openai.OpenAI`（Responses 格式） |
| `anthropic_messages` | 原生 Anthropic Messages API | `anthropic.Anthropic` |

**关键设计**：三种模式在 API 调用前后统一收敛为相同的内部消息格式（OpenAI 风格），切换模型对上层逻辑透明。

### 4.4 工具调用流程

```
模型返回 tool_calls
       │
       ├── 单个工具调用 → 主线程直接执行
       └── 多个工具调用 → ThreadPoolExecutor 并行执行
              │
              ├── 从 registry.py 解析 handler
              ├── 触发 pre_tool_call 插件钩子
              ├── 安全检查（dangerous command 检测）
              ├── 执行 handler
              ├── 触发 post_tool_call 插件钩子
              └── 结果按原始顺序插入对话历史
```

### 4.5 容错与故障转移

```
主模型调用
    │
    ├── 成功 → 继续
    └── 失败（429/5xx/401）
          │
          ├── 检查 fallback_providers 列表
          ├── 按顺序尝试每个备用 Provider
          ├── 401/403 时先尝试凭证刷新
          └── 成功后继续对话（用户无感知）
```

**迭代预算**：默认 90 次迭代上限，达到 100% 时自动停止并返回工作总结。子 Agent 默认上限 50 次。

## 五、模块二：Prompt 系统

### 5.1 三层提示词架构

系统提示词不是扁平的一大段文本，而是**分层有序组装**：

```
┌────────────────────┐
│  stable（稳定层）    │  ← 身份定义、工具引导、Skills 索引
├────────────────────┤       会话期间不变，可被缓存
│  context（上下文层） │  ← 上下文文件（项目规范、README 等）
├────────────────────┤
│  volatile（易变层）  │  ← 记忆内容（MEMORY.md/USER.md）、时间戳
└────────────────────┘       会话期间可能变化
```

**设计原则**：`stable` 层极少变化，可以被 Anthropic 的 Prompt Caching 缓存，降低延迟和成本。

### 5.2 Prompt 缓存策略

- **缓存基础**：系统提示词来自稳定数据源，前缀在多轮对话中基本不变
- **缓存击穿条件**（任一种都会触发全量重编译）：
  1. 会话中途切换模型（`/model` 命令）
  2. 修改记忆文件（`MEMORY.md` 或 `USER.md`）
  3. 修改上下文文件

### 5.3 上下文压缩

当对话长度超过模型上下文的 50% 时，触发**预检压缩**：

```
压缩流程：
1. 先将内存刷新到磁盘（防止数据丢失）
2. 中间对话轮次被辅助 LLM 压缩为精简摘要
3. 最后 N 条消息完整保留（默认 20 条）
4. 工具调用/结果对永不拆分
5. 生成新的会话血统 ID（parent → child 关系）
```

**血统链**：原始对话保留在 SQLite 中，压缩后可追溯更早的上下文，不会永久丢失。

## 六、模块三：Provider 运行时解析

### 6.1 职责

将 `(provider, model)` 元组映射为 `(api_mode, api_key, base_url)`，支持 18+ 个 Provider、OAuth 流程、凭证池和别名解析。

### 6.2 模式解析优先级

```
1. 显式 api_mode 构造参数（最高优先级）
2. 特定 Provider 检测（如 anthropic → anthropic_messages）
3. Base URL 启发式检测（如 api.anthropic.com → anthropic_messages）
4. 默认：chat_completions
```

### 6.3 设计亮点

- **单点接入**：CLI、Gateway、Cron、ACP、辅助调用共享同一个运行时解析器
- **Provider 无关**：切换模型只需改配置，Agent 逻辑零变化
- **国内友好**：任何 OpenAI 兼容端点（DeepSeek、Qwen、智谱等）都可以直接接入

## 七、模块四：工具系统

### 7.1 架构

```
tools/registry.py（中心注册表，70+ 工具，28 个工具集）
       ↑ 每个工具文件在 import 时调用 registry.register() 自注册
       │
   ┌───┼──────┬──────────┬───────────┬─────────────┐
   │   │      │          │           │             │
terminal file   web     browser    code_exec    delegate
(6后端) (5工具) (4后端) (10工具)   (沙箱)     (子Agent)
```

### 7.2 工具自注册机制

```python
# 每个 tool/*.py 文件在顶层调用：
registry.register(
    name="web_search",
    description="Search the web for information",
    parameters={...},
    handler=web_search_handler
)
```

**关键意义**：新增工具只需创建 `tools/` 目录下的文件并调用 `register()`，无需修改任何其他代码，自动发现。

### 7.3 五大工具类别

| 类别 | 工具示例 |
|------|---------|
| **执行类** | 终端命令（6 种后端）、代码运行沙箱 |
| **网页类** | web_search、web_extract、浏览器自动化（10 个工具） |
| **媒体类** | 视觉理解、text-to-image、TTS |
| **协同类** | delegate_task（子Agent调度）、多模型切换 |
| **记忆/规划类** | todo、memory（操作 MEMORY.md）、session_search |

### 7.4 Agent 级工具（特殊拦截）

以下工具在到达 `handle_function_call()` 前被 `run_agent.py` 直接拦截：

| 工具 | 作用 |
|------|------|
| `todo` | 读写 Agent 本地任务状态 |
| `memory` | 向持久化记忆文件写入 |
| `session_search` | 通过 SQLite + FTS5 查询会话历史 |
| `delegate_task` | 生成具有隔离上下文的子 Agent |

## 八、模块五：自进化学习循环

这是 Hermes 区别于所有其他 Agent 框架的**核心差异化能力**——不是靠人手动写规则和流程，而是 AI 自己从实战中总结提升。

### 8.1 四大子模块

```
自进化学习循环
│
├── 1. 自动精选记忆
│   └── Agent 在会话中自动复盘，自己决定什么值得记住
│
├── 2. 自主生成技能
│   └── 满足触发条件时，自动创建技能文件到 ~/.hermes/skills/
│
├── 3. 技能自我进化
│   └── 通过 skill_manage 工具对已有技能进行修补、编辑
│
└── 4. FTS5 会话检索 + LLM 摘要
    └── 先检索相关历史会话，再 LLM 摘要后注入当前上下文
```

### 8.2 技能自动生成触发条件

满足以下**任一条件**，Agent 会主动创建技能文件：

| 触发条件 | 说明 |
|----------|------|
| 调用了 **5 次及以上**工具 | "做了不少事，值得记下来" |
| 从错误里**成功恢复** | "这次踩坑下次别踩" |
| 用户给了**修正指导** | "用户教过我怎么做" |
| 蹚通了一套**不那么直观的有效流程** | "这套流程不容易想到，值得固化" |

### 8.3 技能文件结构（agentskills.io 标准）

```yaml
---
name: deploy-to-staging
description: 将代码部署到预发布环境的完整流程
version: 1.0.0
platforms: [linux, macos]
metadata:
  hermes:
    tags: [deployment, devops]
    category: devops
    fallback_for_toolsets: [terminal]
    requires_toolsets: [terminal]
---
```
技能文件包含完整的执行步骤、工具调用序列、文件引用等，可跨兼容 Agent 直接迁移。

### 8.4 技能管理与进化

通过 `skill_manage` 工具支持 6 种操作：

| 操作 | 说明 |
|------|------|
| `create` | 创建技能 |
| `patch` | **补丁修改（默认推荐）**——只传旧文本和替换内容 |
| `edit` | 编辑技能 |
| `delete` | 删除技能 |
| `write_file` | 向技能写文件 |
| `delete_file` | 从技能删文件 |

**关键设计**：默认用 `patch` 而非全量重写，原因：
1. 保证正确性（全量重写可能把好用的部分改崩）
2. 省 Token
3. 执行更轻量

## 九、模块六：四层内存系统

Hermes 的记忆不是简单的"保存对话历史"，而是分四层独立管理：

```
┌────────────────────────────────────────────────────────────────┐
│                  四层内存系统                                    │
│                                                                 │
│  ┌──────────────────────────────────────┐                      │
│  │ Layer 1: Prompt 记忆（常驻层）         │ 自动加载，每次会话   │
│  │ MEMORY.md / USER.md                  │ 限制 3575 字符       │
│  │ ~/.hermes/memories/                  │                      │
│  └──────────────────────────────────────┘                      │
│  ┌──────────────────────────────────────┐                      │
│  │ Layer 2: 会话检索（主动调取层）        │ 按需检索，命中后加载  │
│  │ SQLite + FTS5                        │                      │
│  │                                      │                      │
│  └──────────────────────────────────────┘                      │
│  ┌──────────────────────────────────────┐                      │
│  │ Layer 3: 技能记忆（过程记忆层）         │ 渐进式披露           │
│  │ ~/.hermes/skills/                    │ 默认只加载名称+简介   │
│  │ 独立 Markdown 文件                    │ 需要时才加载完整内容  │
│  └──────────────────────────────────────┘                      │
│  ┌──────────────────────────────────────┐                      │
│  │ Layer 4: Honcho 层（用户建模）         │ 可选层               │
│  │ 跨会话用户画像、偏好、风格             │ 适合长期私人助理     │
│  └──────────────────────────────────────┘                      │
└────────────────────────────────────────────────────────────────┘
```

### 各层详细说明

| 层级 | 存储位置 | 加载时机 | 内容 | 用途 |
|------|---------|---------|------|------|
| **Layer 1** | `~/.hermes/memories/` | 会话开始自动加载 | 关键偏好、习惯、规则 | 每次对话都需知道的信息 |
| **Layer 2** | SQLite + FTS5 | Agent 主动按需检索 | 过往会话的完整记录 | 需要回顾历史时 |
| **Layer 3** | `~/.hermes/skills/` | 渐进式披露（先名称+简介） | 已验证有效的工作流程 | "这事儿我以前做过" |
| **Layer 4** | Honcho 引擎 | 跨会话幕后建模 | 用户画像、说话风格 | 让回复风格更贴合你 |

### 渐进式技能加载（关键设计）

```
200 个技能
    │
    ├── 默认加载：只加载技能名称 + 简介（<200 Tokens 总量）
    │                                       ↑ 几乎不随技能数量增长
    └── 按需加载：Agent 判断需要时才加载完整内容
```

## 十、模块七：网关模块

### 10.1 定位

一个**常驻后台服务**，负责将 Hermes Agent 接入各种消息平台，让 Agent 可以在 Telegram、飞书、钉钉、WhatsApp 等 20+ 平台上使用，且跨平台共享记忆和会话上下文。

### 10.2 支持的平台

```
gateway/platforms/
├── telegram       ├── signal        ├── wecom（企业微信）
├── discord        ├── matrix        ├── wecom_callback
├── slack          ├── mattermost    ├── weixin（微信）
├── whatsapp       ├── email         ├── qqbot（QQ机器人）
├── dingtalk       ├── sms           ├── homeassistant
│   （钉钉）        │                │
├── feishu         ├── bluebubbles   ├── webhook
│   （飞书）        │                │
│                  │                ├── api_server
│                  │                └── yuanbao（腾讯元宝）
```

### 10.3 核心设计

- **统一会话路由**：会话绑定的是用户 ID，不是平台——在 Telegram 开头的话，去终端可以接着聊
- **用户授权**：白名单 + DM 配对双重机制
- **钩子系统**：消息生命周期的各个阶段（接收、处理、发送前后）都有钩子可扩展
- **镜像投递**：跨会话同步消息

### 10.4 与其他框架网关的关键区别

| 对比维度 | 其他框架的网关 | Hermes 网关 |
|----------|-------------|-----------|
| 职责范围 | 只管消息收发 | 网关是学习循环的一部分 |
| 技能生成 | 不与网关打通 | 网关收到的消息就能触发技能生成 |
| 定时任务输出 | 走独立通道 | 结果通过同一层网关回传给用户 |
| 跨平台体验 | 后期补丁式拼接 | 从设计之初就内嵌在系统里 |

## 十二、模块九：Cron 定时自动化

### 12.1 定位

Hermes 将定时任务视为**一等公民 Agent 任务**，而非简单的 Shell 脚本。

### 12.2 工作流程

```
设置定时消息 → 解析指令 → 存储到 cron/ 目录（JSON 格式）
                                  │
                    网关 tick 机制定时触发
                                  │
                    Agent 以完整权限执行
                    （可调用工具、读取记忆、使用技能）
                                  │
                    结果通过网关投递到目标平台
                                  │
                    更新任务状态和 next_run
```

**关键区别**：不是把 AI 当 Shell 脚本用，而是让 Agent 带着它的全部能力（记忆、技能、工具）去执行定时任务，结果走完整的网关投递链路。

### 14.1 技术

| 项目 | 详情 |
|------|------|
| 存储引擎 | SQLite（便携文件型，零外部依赖） |

### 14.2 会话血统追踪

```
原始会话（parent）
    │
    ├── 触发压缩
    │
    └── 子会话（child）
        ├── 保留压缩摘要
        └── 可通过 parent_id 追溯到原始对话
```

### 14.3 Profile 隔离

每个 Profile（`hermes -p <name>`）拥有独立的：`HERMES_HOME`、配置、记忆、会话、Gateway PID，多个 Profile 可并行运行互不干扰。

## 十六、完整工作流示例

下面用一个**完整示例**展示 Hermes Agent 处理一个真实任务的全链路。假设用户通过 CLI 发送了一条消息。

### 场景

> 用户在终端输入：`"帮我把 data.csv 里的销售数据按月份统计，生成一张趋势图，然后发到飞书群里"`

### 全链路执行过程

```
═══════════════════════════════════════════════════════════════════
步骤 1：入口接收
═══════════════════════════════════════════════════════════════════

CLI (cli.py) 接收用户输入
  → HermesCLI.process_input("帮我把 data.csv 里的...")
  → 调用 AIAgent.run_conversation()

═══════════════════════════════════════════════════════════════════
步骤 2：初始化
═══════════════════════════════════════════════════════════════════

AIAgent.run_conversation()
  ├── 生成 task_id = "task_abc123"
  ├── 追加用户消息到 conversation_history：
  │   [{"role": "user", "content": "帮我把 data.csv 里的..."}]
  │
  ├── 调用 prompt_builder.build_system_prompt()
  │   ├── 加载 Layer 1 记忆：MEMORY.md → "用户偏好中文回复，喜欢简洁风格"
  │   ├── 加载 Layer 1 记忆：USER.md → "数据科学家，常用 pandas/matplotlib"
  │   ├── 加载 Layer 3 技能索引（只加载名称+简介）：
  │   │   ├── data-analysis → "数据分析与可视化流程"
  │   │   └── feishu-notify → "向飞书群发送消息"
  │   └── 组装三层提示词：stable → context → volatile
  │
  ├── 通过 runtime_provider 解析 Provider：
  │   (provider="deepseek", model="deepseek-chat") 
  │   → (api_mode="chat_completions", base_url="https://api.deepseek.com/v1")
  │
  └── 预检压缩：当前上下文 8K / 模型窗口 64K = 12.5% < 50%，不触发压缩

═══════════════════════════════════════════════════════════════════
步骤 3：第一轮 LLM 调用
═══════════════════════════════════════════════════════════════════

  → _interruptible_api_call（可中断的 API 调用）
  → 发送给 DeepSeek API
  → 模型返回 tool_calls：

═══════════════════════════════════════════════════════════════════

  → todo 工具被 AIAgent 层拦截（Agent 级工具）
    └── 直接修改 Agent 内部任务状态，不通过 registry

  → read_file 进入正常工具执行流程：
    ├── 从 tools/registry.py 解析 handler
    ├── 触发 pre_tool_call 钩子
    ├── 安全检查：不是 dangerous command，跳过
    ├── 执行 file_tools.read_file_handler("data.csv")
    ├── 触发 post_tool_call 钩子
    └── 返回结果 → 追加到对话历史
```

## 十七、设计哲学总结

| 设计原则 | 具体含义 | 对用户的价值 |
|----------|---------|------------|
| **Prompt 稳定性** | 系统提示词对话中途不变 | 缓存命中率高，省 Token 省延迟 |
| **可观测执行** | 每次工具调用可见（spinner/进度消息） | 用户知道 Agent 在做什么 |
| **可中断** | API 调用/工具可中途取消 | 不会"卡死"，随时接手 |
| **平台无关核心** | 一个 AIAgent 类服务所有入口 | 逻辑不会因入口不同而分裂 |
| **松耦合** | 可选子系统用注册表+check_fn 门控 | 按需启用，不用的零开销 |
| **Profile 隔离** | 每个 Profile 独立配置/记忆/会话 | 工作和个人完全隔离 |
| **自我进化** | Agent 自己从实战中学习 | 越用越聪明，不需要手动调优 |
| **优先补丁** | 更新技能时用 patch 而非全量重写 | 保证正确性，省 Token |
| **渐进式披露** | 技能默认只加载简介，用时才加载详情 | 技能再多成本几乎不变 |
