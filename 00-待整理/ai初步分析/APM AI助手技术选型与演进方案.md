# 1 APM AI助手技术选型与演进方案

> 适用日期：2026-06-29  
> 面向团队：3名Python开发人员，具备APM后端与数据层经验，但AI/大模型工程经验较弱  
> 核心建议：采用 **LangGraph + 受控数据工具层 + 前端ECharts图表协议**，先实现可解释的异常分析闭环，再逐步扩展为多数据源、多Agent、可灰度上线的智能诊断系统。

---

## 1.1 背景与目标

APM系统已经具备交换机性能与健康数据采集、异常检测、指标图表展示和异常告警能力。现阶段目标是在Web端新增AI助手，使用户可以通过自然语言完成异常分析，例如：

> “我的端口为什么丢包？”

AI助手需要完成从问题理解、数据规划、多源查询、根因推断、证据整理到图表生成的闭环。

系统现有数据源包括：

| 数据源 | 主要数据 | AI助手接入目标 |
|---|---|---|
| MySQL | 设备元数据、端口信息、告警事件、配置、用户权限 | 查询设备上下文、历史告警、配置变更、权限校验 |
| Redis | 实时指标缓存、会话状态、热数据 | 获取最新状态、缓存AI中间结果、维护会话上下文 |
| VictoriaMetrics | CPU、内存、端口丢包、端口缓存等大规模时序指标 | 查询趋势、对比、异常时间段指标 |
| Pyroscope | 进程级持续性能剖析数据、CPU火焰图 | 分析CPU热点、进程资源异常、性能瓶颈 |
| Loki | 交换机日志、系统日志、运行日志 | 按时间窗口关联日志、定位错误、配置变更、链路异常 |

AI助手不应直接暴露数据库给大模型，而应通过受控工具函数访问数据源。

### 1.2.2 关键维度分析

#### 1.2.2.1 LangGraph

LangGraph更像一个Agent工作流引擎，而不是简单的聊天框架。它适合把复杂诊断任务拆成多个可控节点，例如：

1. 理解用户问题。
2. 识别设备、端口、时间范围。
3. 选择诊断Playbook。
4. 调用指标、日志、告警、剖析工具。
5. 汇总证据。
6. 生成根因分析。
7. 输出图表描述。

它的优势是流程可控、状态清晰、便于调试和审计。APM异常诊断本质上是多步骤、有状态、可能失败、需要降级的工程流程，LangGraph最贴合。

#### 1.2.2.2 CrewAI

CrewAI的优点是上手快，角色和任务模型直观，适合团队快速做Demo。例如可以定义：

- 规划Agent
- 指标查询Agent
- 日志分析Agent
- 报告Agent

但CrewAI偏高层抽象，复杂控制流、失败重试、局部降级、工具调用审计等能力需要额外设计。对于生产级APM系统，后期可能需要更多工程约束。

#### 1.2.2.3 AutoGen

AutoGen适合多Agent对话、研究型协作和动态任务分解。但APM场景需要强边界、强权限控制、强审计。若使用AutoGen，需要花较多精力限制Agent发散式对话。

#### 1.2.2.4 MetaGPT

MetaGPT更适合把固定SOP转成多角色产出流程，例如需求分析、架构设计、代码生成。APM在线异常诊断需要实时查询、多源聚合、用户权限控制和低延迟交互，因此MetaGPT不是首选。

## 1.3 框架选型建议

### 1.3.1 推荐选型

推荐采用：

- **Agent框架：LangGraph**
- **后端语言：Python**
- **服务形态：独立AI Assistant Service，与现有APM后端通过内部API集成**
- **模型：商用API，优先选择支持工具调用、JSON Schema/结构化输出、流式输出的模型**
- **数据访问：自研受控工具层**
- **图表方案：Agent输出结构化图表描述，前端ECharts渲染**

### 1.3.2 推荐理由

#### 1.3.2.1 APM异常分析是强流程任务

“端口为什么丢包”不能只靠一次问答完成。系统至少要分析：

- 哪台设备？
- 哪个端口？
- 哪个时间窗口？
- 丢包是否真实升高？
- 是否伴随端口缓存升高？
- 是否伴随流量突增？
- 是否伴随系统CPU或内存异常？
- 是否有相关日志？
- 是否有历史告警或配置变更？
- 是否有进程级性能瓶颈？

LangGraph可以把这些步骤固化成可控流程，而不是让大模型自由发挥。

#### 1.3.2.2 多数据源必须通过工具层安全接入

不建议让LLM直接生成并执行：

- SQL
- Redis命令
- MetricsQL/PromQL
- LogQL
- Pyroscope查询语句

正确做法是封装成受控工具：

```python
query_port_drop_rate(device_id, port_id, start_time, end_time, step)
query_device_alerts(device_id, start_time, end_time, severity)
query_loki_logs(device_id, keywords, start_time, end_time, limit)
query_profile_top_functions(device_id, process_name, start_time, end_time)
```

LLM只负责选择工具和填写参数，工具内部负责权限校验、查询模板、超时限制、脱敏和审计。

#### 1.3.2.3 团队Python熟练但AI经验较弱

LangGraph学习成本略高于CrewAI，但长期收益更明显。建议团队不要一开始做复杂自治多Agent，而是从最小图开始：

```text
Router -> Planner -> Tool Executor -> Analyzer -> Visualizer
```

逐步增加Agent数量和诊断Playbook。

#### 1.3.2.4 图表生成应复用现有Web能力

Agent不应生成可执行JS代码，也不应把Python绘图代码交给线上执行。推荐输出安全JSON：

```json
{
  "type": "line",
  "title": "端口丢包率趋势",
  "xAxis": {
    "type": "time"
  },
  "yAxis": {
    "name": "drop_rate"
  },
  "series": [
    {
      "name": "丢包率",
      "data": [
        ["2026-06-29T10:00:00+08:00", 0.01],
        ["2026-06-29T10:05:00+08:00", 0.08]
      ]
    }
  ]
}
```

前端只渲染经过JSON Schema校验的安全字段。

### 1.3.3 备选方案

| 场景 | 备选 |
|---|---|
| 只做1-2周内部Demo | CrewAI |
| 研究多Agent自由协作能力 | AutoGen |
| 生成运维报告、故障复盘文档 | MetaGPT或普通LangChain链路 |
| 正式生产APM AI助手 | LangGraph |

## 1.4 开发难点识别与应对思路

### 1.4.1 复杂分析任务的规划与编排

**难点：**

用户问题通常不完整，例如“为什么丢包”，可能缺少设备、端口、时间范围。Agent需要自动判断需要哪些数据，并按合理顺序查询。

**风险：**

- 漏查关键指标。
- 查询顺序混乱。
- 过度查询导致延迟高。
- 大模型编造分析步骤。

**应对思路：**

建立固定诊断Playbook，而不是完全让模型自由规划。

示例：端口丢包诊断Playbook

1. 解析设备、端口、时间窗口。
2. 查询端口丢包率趋势。
3. 查询端口流量、错误包、端口缓存利用率。
4. 查询设备系统CPU、内存。
5. 查询相关告警。
6. 查询同时间窗口Loki日志。
7. 必要时查询Pyroscope进程热点。
8. 汇总证据，输出根因候选和置信度。

### 1.4.2 不同数据库的安全工具设计与调用

**难点：**

自然语言到查询语句的转换容易出现错误、注入、越权、慢查询。

**应对思路：**

| 数据源 | 工具设计建议 |
|---|---|
| MySQL | 不开放任意SQL；提供领域工具，如`get_device_info`、`list_alerts`、`get_config_changes`；只读账号；租户过滤 |
| Redis | 不开放原生命令；仅允许访问AI命名空间或白名单key；写入必须带TTL |
| VictoriaMetrics | 查询模板化；限制时间范围、step、max_points；默认降采样 |
| Pyroscope | 不返回完整profile给LLM；返回top函数、CPU热点摘要、profile链接 |
| Loki | 固定label和关键词模板；限制返回条数；日志脱敏 |

### 1.4.3 高延迟查询下的用户体验保障

**难点：**

Pyroscope、Loki、大范围时序查询可能耗时较长。

**应对思路：**

- 使用SSE或WebSocket流式返回进度。
- 查询任务异步化。
- 先返回初步结论，再补充深度结果。
- 工具层设置超时和降级。
- 大范围查询自动降采样。
- 对相同诊断查询做短期缓存。

前端进度示例：

```text
已识别设备 SW-01，端口 Gi1/0/24
正在查询最近2小时丢包率趋势
正在关联端口缓存利用率
正在检索同时间窗口日志
正在生成诊断报告
```

### 1.4.4 多源数据聚合与关联分析

**难点：**

不同数据源的时间戳、设备ID、端口名、指标名称可能不一致。

**应对思路：**

建立统一Evidence Schema：

```json
{
  "evidence_id": "ev_001",
  "source": "victoriametrics",
  "entity": {
    "device_id": "SW-01",
    "interface_id": "Gi1/0/24"
  },
  "time_window": {
    "start": "2026-06-29T10:00:00+08:00",
    "end": "2026-06-29T12:00:00+08:00"
  },
  "metric": "port_drop_rate",
  "summary": {
    "max": 0.12,
    "avg": 0.03,
    "baseline": 0.005
  }
}
```

分析Agent只能基于Evidence输出结论，报告中必须引用证据ID。

### 1.4.5 图表生成与前端集成

**难点：**

直接执行Agent生成的JS或Python代码存在安全风险。

**应对思路：**

- Agent只输出结构化图表描述。
- 后端使用JSON Schema校验。
- 前端只支持安全ECharts option子集。
- 禁止函数字符串、HTML、外部脚本URL。
- 图表数据必须来自工具查询结果，而不是模型编造。

推荐图表类型：

| 图表类型 | 用途 |
|---|---|
| 折线图 | 丢包率、CPU、内存、缓存利用率趋势 |
| 双轴折线图 | 丢包率与端口流量、缓存利用率对比 |
| 柱状图 | Top N端口丢包、Top N进程CPU |
| 表格 | 告警列表、日志摘要、根因候选 |
| 火焰图链接或嵌入组件 | Pyroscope profile结果 |

### 1.4.6 异常处理与降级策略

**难点：**

某个数据源不可用时，不能让整个AI助手失败。

**应对思路：**

工具返回统一状态：

```json
{
  "status": "partial",
  "source": "loki",
  "message": "日志查询超时，仅返回前100条结果",
  "data": []
}
```

报告中明确说明：

- 哪些数据已查询。
- 哪些数据缺失。
- 当前结论置信度。
- 建议用户补充哪些信息或扩大时间范围。

### 1.4.7 与现有APM后端和Web前端集成

**难点：**

AI能力不应侵入原有核心业务代码，也不应绕过现有权限体系。

**应对思路：**

推荐新增独立服务：

```text
apm-ai-assistant-service
```

该服务通过内部API或SDK调用现有APM后端能力。所有请求必须携带：

- user_id
- tenant_id
- role
- device_scope
- session_id

前端新增：

- 聊天面板
- 诊断报告卡片
- 图表卡片
- 工具调用进度
- 用户反馈入口

### 1.4.8 提示词工程与多智能体协作策略

**难点：**

多Agent容易互相重复、发散、过度推理，甚至生成没有证据的结论。

**应对思路：**

- 每个Agent职责固定。
- 所有输出结构化。
- 禁止无证据结论。
- 根因必须引用Evidence。
- Planner只能选择预定义Playbook。
- Analyzer必须输出置信度和缺失证据。

Agent角色建议：

| Agent | 职责 |
|---|---|
| Planner Agent | 识别意图、实体、时间窗口，选择诊断Playbook |
| Metric Agent | 查询VictoriaMetrics指标 |
| Metadata Agent | 查询MySQL设备、端口、告警、配置 |
| Realtime Agent | 查询Redis实时状态 |
| Log Agent | 查询Loki日志并摘要 |
| Profiling Agent | 查询Pyroscope性能剖析摘要 |
| Analysis Agent | 汇总证据并推断根因 |
| Visualization Agent | 生成安全chart_spec |

### 1.4.9 安全与权限控制

**难点：**

AI助手可能成为越权查询入口。

**应对思路：**

- LLM不接触数据库账号和连接串。
- 工具层统一鉴权。
- 所有查询按租户和设备权限过滤。
- 敏感字段脱敏。
- 全量审计工具调用。
- 限制查询时间范围、返回条数、调用频率。
- 写操作只允许写AI会话、反馈、审计、缓存，不允许直接改设备配置。

## 1.6 分阶段演进方案

### 1.6.1 阶段0：技术预研与架构骨架

| 项目 | 内容 |
|---|---|
| 阶段目标 | 跑通“用户提问 -> Agent规划 -> 工具调用 -> 结构化回答”的最小闭环 |
| 核心功能 | 模型API接入；LangGraph最小状态图；工具调用日志；基础聊天接口 |
| 数据源范围 | 暂不接全部数据源，可先Mock或只接MySQL设备元数据 |
| Agent数量与职责 | 1-2个：Router/Planner负责问题理解，Answer Agent负责生成回答 |
| 关键交付物 | AI服务骨架、工具注册规范、统一返回schema、审计日志表设计 |
| 预估耗时 | 1-2周 |

### 1.7.2 端到端交互流程

```text
1. 用户在Web端提问：
   “我的端口为什么丢包？”

2. Web前端发送请求到APM后端：
   question + 当前页面设备上下文 + 用户身份 + 租户信息

3. APM后端完成鉴权后调用AI Assistant Service。

4. Planner Agent解析：
   - 意图：端口丢包诊断
   - 实体：设备、端口
   - 时间范围：默认最近2小时，或从用户问题提取
   - Playbook：port_packet_loss_diagnosis

5. Data Agents调用受控工具：
   - MySQL：查设备、端口、告警、配置变更
   - VictoriaMetrics：查丢包率、流量、缓存、CPU、内存
   - Redis：查实时状态
   - Loki：查同时间窗口日志
   - Pyroscope：必要时查进程CPU热点

6. Evidence Hub统一证据：
   - 时间对齐
   - 单位归一
   - 证据ID生成
   - 数据源状态记录

7. Analyzer Agent生成诊断报告：
   - 结论
   - 根因候选
   - 置信度
   - 证据引用
   - 缺失信息
   - 建议动作

8. Visualization Agent生成chart_spec：
   - 丢包率趋势图
   - 缓存利用率对比图
   - CPU/内存趋势图
   - 告警和日志摘要表

9. 前端校验chart_spec并用ECharts渲染。

10. 工具调用、模型调用、用户反馈写入审计和分析记录。
```

## 1.8 数据源工具设计建议

### 1.8.1 工具层基本原则

1. **LLM永远不直接连接数据库。**
2. **LLM不能执行任意SQL、Redis命令、LogQL或MetricsQL。**
3. **所有工具入参必须经过类型校验和权限校验。**
4. **所有工具调用必须记录审计日志。**
5. **所有查询必须设置超时、最大返回条数、最大时间窗口。**
6. **工具返回统一结构，便于Agent聚合。**

### 1.8.2 工具返回标准结构

```json
  "tool_name": "query_port_drop_rate",

    "end": "2026-06-29T12:00:00+08:00"
  },
  "data": {},

  "warnings": []

```

## 1.9 风险与控制措施

| 风险 | 影响 | 控制措施 |
|---|---|---|
| 大模型幻觉 | 错误根因、误导运维 | 结论必须引用Evidence；无证据不得断言 |
| 查询越权 | 数据泄露 | 工具层统一鉴权，按租户和设备过滤 |
| 查询过慢 | 用户体验差 | 异步、流式、超时、缓存、降采样 |
| 成本过高 | 商用API费用不可控 | 小模型做规划，大模型做最终报告；压缩上下文；缓存 |
| 图表安全风险 | XSS或前端执行恶意代码 | 只允许JSON图表协议，禁止执行模型代码 |
| Agent流程不可控 | 重复查询、死循环 | LangGraph设置最大步数、状态机边界和超时 |
| 数据质量不一致 | 根因判断错误 | 统一实体模型、时间对齐、单位归一 |
| 团队AI经验不足 | 进度延期 | 先固定Playbook，避免一开始做自由自治Agent |

## 1.10 最终推荐结论

本项目建议采用 **LangGraph作为Agent编排框架**，以 **受控工具层** 接入MySQL、Redis、VictoriaMetrics、Pyroscope、Loki，以 **结构化chart_spec + 前端ECharts** 实现图表展示。

落地路线应坚持以下原则：

1. **先做窄场景，不做万能助手。**  
   第一阶段只聚焦“端口丢包诊断”。

2. **先做证据链，不做自由发挥。**  
   所有结论必须来自工具返回的证据。

3. **先接MySQL和VictoriaMetrics，再接Redis、Loki，最后接Pyroscope。**

4. **先把图表协议跑通，再扩展复杂可视化。**

5. **所有数据访问都必须走后端受控工具，不能让大模型直连数据库。**

6. **多Agent能力逐步增强，初期以固定Playbook和可控状态机为主。**

按该路线，团队可在 **4-6周** 内交付可演示MVP，在 **15-22周** 内达到生产灰度水平。
