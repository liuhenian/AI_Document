在 LangGraph 里，一般这样理解：

- **短期记忆**：当前线程/会话内，为了本次任务连续推理而保留的上下文  
- **长期记忆**：跨线程、跨会话、跨时间还能取回的用户/业务知识

最关键的一句话：

> **LangGraph 的短期记忆主要靠 state + checkpoint；长期记忆主要靠外部存储 + 检索/写回。**

---

# 1 一、先区分：短期记忆 vs 长期记忆

## 1.1 1）短期记忆
短期记忆通常是：
- 当前对话历史
- 当前任务的中间结果
- 工具调用结果
- 当前计划
- 本轮用户提供的临时信息

特点：
- 生命周期通常是“一个会话 / 一个 thread”
- 主要服务于当前任务连续性
- 可以中断恢复
- 不一定长期保存

在 LangGraph 里，短期记忆最典型就是：
- `messages`
- `state`
- `checkpoint`

---

## 1.2 2）长期记忆
长期记忆通常是：
- 用户画像
- 用户偏好
- 历史重要事实
- 跨会话经验
- 业务知识沉淀
- 历史任务结果摘要

特点：
- 跨会话持久化
- 需要按用户/租户隔离
- 需要检索策略
- 需要更新策略
- 不建议把所有历史原文都塞给模型

在 LangGraph 里，长期记忆通常不是 graph 自动帮你完成的，而是：
- 你自己设计 memory store
- 在图里做“读记忆”和“写记忆”节点

---

# 2 二、LangGraph 里的短期记忆怎么实现？

短期记忆通常有两层：

## 2.1 方案 1：直接放在 State 里
这是最基本的方式。

例如：

```python
from typing import TypedDict, List
from langchain_core.messages import BaseMessage

class AgentState(TypedDict, total=False):
    messages: List[BaseMessage]
    user_input: str
    tool_result: str
    final_answer: str
```

在节点之间不断更新 `messages`、`tool_result`、`plan` 等，就是短期记忆。

### 2.1.1 适合
- 单次 invoke
- 简单多轮上下文
- 当前任务中间变量传递

### 2.1.2 优点
- 简单直接
- 节点可见
- 容易调试

### 2.1.3 缺点
- 只在这次执行里有效
- 如果没有持久化，进程结束就没了

---

## 2.2 方案 2：用 Checkpointer 做线程级短期记忆
这是 LangGraph 更正式的做法。

你可以把图编译时挂上 checkpointer，这样每次执行都会把 state 存下来。  
下次同一个 `thread_id` 再调用时，可以接着之前的上下文继续。

### 2.2.1 作用
- 保存当前 thread 的对话状态
- 支持多轮会话
- 支持中断恢复
- 支持“会话级记忆”

### 2.2.2 常见 checkpointer
- `MemorySaver`：内存版，适合开发测试
- SQLite / Postgres 等持久化 saver：适合生产

---

## 2.3 短期记忆示意思路

```python
from langgraph.checkpoint.memory import MemorySaver

checkpointer = MemorySaver()
graph = builder.compile(checkpointer=checkpointer)

graph.invoke(
    {"messages": [...]},
    config={"configurable": {"thread_id": "user_001"}}
)
```

之后只要还用同一个 `thread_id`，图就能拿到这个线程之前的 state。

### 2.3.1 这里的本质
- `thread_id` = 会话标识
- checkpointer = 当前会话状态存储器
- state = 这个会话的短期记忆内容

---

# 3 三、LangGraph 里的长期记忆怎么实现？

长期记忆一般不直接靠 checkpoint。

因为 checkpoint 更偏向：
- 执行状态
- 会话状态
- 恢复现场

而长期记忆更偏向：
- 用户事实
- 偏好
- 历史知识
- 可长期复用的信息

所以长期记忆通常要你自己接外部存储。

---

## 3.1 长期记忆常见存储方式

### 3.1.1 1）关系型数据库
适合存：
- 用户画像
- 偏好
- 配置
- 历史标签
- 结构化事实

比如：
- MySQL
- PostgreSQL
- SQLite

#### 3.1.1.1 示例存什么
- 用户喜欢中文回复
- 用户常用项目是 A 系统
- 用户所在地区
- 用户历史订单偏好

---

### 3.1.2 2）向量数据库
适合存：
- 语义检索型历史记忆
- 历史对话摘要
- 重要片段
- 历史经验库

比如：
- Chroma
- FAISS
- Milvus
- PGVector
- Weaviate
- Qdrant

#### 3.1.2.1 适合场景
你想根据当前问题，从历史里“召回相关记忆”。

例如：
- 用户曾说“我不喜欢太正式的语气”
- 这次提问时检索出这个偏好

---

### 3.1.3 3）KV / 文档数据库
适合存：
- 用户 profile
- session summary
- memory list
- JSON 结构数据

比如：
- Redis
- MongoDB
- DynamoDB

---

# 4 四、长期记忆的标准实现方式

通常不是“每轮都把所有历史扔给模型”，而是：

## 4.1 记忆写入
在某些节点判断：
- 这条信息值不值得长期保存？
- 需要保存成什么形式？

例如抽取出：
- 用户偏好
- 长期事实
- 重要任务结果

然后写入数据库/向量库。

---

## 4.2 记忆读取
每次新问题来了，先根据：
- user_id
- 当前 query
- 场景标签

去查长期记忆。

查到后把相关记忆注入 state，例如：
- `long_term_memories`
- `user_profile`
- `retrieved_memory`

再让后续节点生成答案。

---

## 4.3 记忆压缩/更新
长期记忆不能无限长，所以要做：
- 去重
- 合并
- 摘要
- 过期
- 优先级管理

---

# 5 五、一个推荐的架构：短期记忆 + 长期记忆结合

这是最常见、也是最实用的模式。

## 5.1 短期记忆负责：
- 当前 thread 的多轮上下文
- 当前任务中间状态
- 工具结果
- 当前会话 messages

## 5.2 长期记忆负责：
- 跨会话用户事实
- 偏好
- 历史摘要
- 可复用经验

---

## 5.3 典型流程

```text
用户输入
   ↓
读取长期记忆（用户画像/历史偏好/相关历史）
   ↓
写入当前 state
   ↓
结合当前短期记忆（messages/checkpoint）
   ↓
LLM 决策 / 工具调用 / 回答
   ↓
从本轮对话中抽取值得长期保存的信息
   ↓
写回长期记忆库
```

---

# 6 六、短期记忆具体怎么做

下面给你一个更具体的概念。

## 6.1 方法一：messages 作为短期记忆
最常见：

```python
class AgentState(TypedDict, total=False):
    messages: list
```

每轮都把：
- 用户消息
- AI 消息
- 工具消息

加入 `messages`。

### 6.1.1 优点
- 自然兼容聊天模型
- 简单

### 6.1.2 问题
- 越来越长
- token 成本高
- 可能需要裁剪或摘要

---

## 6.2 方法二：把中间变量也当短期记忆
除了 `messages`，还可以保留：

```python
class AgentState(TypedDict, total=False):
    messages: list
    current_plan: str
    retrieved_docs: list
    tool_result: str
    iteration_count: int
```

这些都属于短期工作记忆。

---

## 6.3 方法三：配合 checkpoint
通过 `thread_id` 把上述 state 绑定到当前会话。

这时你能得到：
- 多轮连续
- 中断恢复
- 会话追踪

---

# 7 七、长期记忆具体怎么做

长期记忆推荐拆成两种：

## 7.1 1）Profile 型记忆
结构化、稳定。

例如：
```json
{
  "user_id": "u001",
  "language": "zh",
  "tone_preference": "简洁",
  "industry": "电商",
  "common_goal": "订单自动化处理"
}
```

适合存在数据库。

---

## 7.2 2）Episodic 型记忆
事件型、片段型、经验型。

例如：
- “2025-06-01 用户说他更关注退款流程”
- “用户在上次项目中使用的是 PostgreSQL”
- “用户不希望模型输出太长”

这种可以：
- 存成文本片段
- 做 embedding
- 放向量库检索

---

# 8 八、长期记忆不是“存全部对话原文”

这是一个非常重要的实践点。

不要简单地：
- 把每次所有聊天记录都原封不动存成长期记忆
- 然后每次全量取回来

这样会有几个问题：
- 噪音太多
- token 太大
- 相关性差
- 记忆污染
- 隐私风险

更好的方式是：
- 抽取“重要事实”
- 抽取“稳定偏好”
- 抽取“有复用价值的信息”
- 存成结构化或摘要化内容

---

# 9 九、在 LangGraph 中一般如何落地记忆节点？

你可以在图里显式加两个节点：

## 9.1 1）memory_read_node
做的事：
- 根据 `user_id` 和当前 query
- 从 DB / 向量库读取相关长期记忆
- 写入 state

例如写入：
```python
{
  "user_profile": {...},
  "retrieved_memory": [...]
}
```

---

## 9.2 2）memory_write_node
做的事：
- 从当前会话结果中抽取值得保存的事实
- 写入长期记忆系统

例如：
- 用户偏好
- 重要项目事实
- 关键结论

---

# 10 十、一个实际推荐的 State 设计

比如：

```python
from typing import TypedDict, List, Dict, Any
from langchain_core.messages import BaseMessage

class AgentState(TypedDict, total=False):
    user_id: str
    messages: List[BaseMessage]

    # 短期记忆
    current_task: str
    tool_result: str
    retrieved_docs: List[str]
    final_answer: str

    # 长期记忆
    user_profile: Dict[str, Any]
    retrieved_memory: List[str]

    # 控制字段
    next_step: str
    error: str
```

---

# 11 十一、记忆更新策略怎么定？

长期记忆最重要的不是“能存”，而是“存什么”。

建议你定义规则：

## 11.1 适合写入长期记忆的内容
- 用户稳定偏好
- 身份/角色/组织信息
- 长期目标
- 重复提及的重要事实
- 高价值历史结论
- 已确认的业务事实

## 11.2 不适合写入长期记忆的内容
- 一次性临时信息
- 未确认的信息
- 情绪化噪音
- 敏感但无必要的信息
- 冗长中间推理过程

---

# 12 十二、短期记忆也需要管理

短期记忆不是无限堆 messages。

你要考虑：

## 12.1 1）裁剪
只保留最近 N 轮

## 12.2 2）摘要
把旧对话压缩成 summary

## 12.3 3）分层
- recent messages
- conversation summary
- current tool results

这是比“全量 messages”更稳的做法。

---

# 13 十三、生产实践建议

## 13.1 对短期记忆
建议：
- 用 checkpointer
- 每个用户/会话一个 `thread_id`
- messages 做裁剪或摘要
- 重要中间变量写入 state

## 13.2 对长期记忆
建议：
- 不要依赖 checkpoint 充当长期记忆
- 用外部数据库/向量库
- 设计 memory read / memory write 节点
- 做用户隔离、租户隔离、权限控制
- 只保存高价值信息

---

# 14 十四、一个简单结论

## 14.1 短期记忆
在 LangGraph 中主要靠：

- `State`
- `messages`
- `checkpointer`
- `thread_id`

## 14.2 长期记忆
在 LangGraph 中主要靠：

- 外部数据库 / 向量库 / KV
- 记忆读取节点
- 记忆写入节点
- 摘要 / 抽取 / 检索策略

# 15 十五、最推荐的实现思路

如果你现在准备做项目，我建议你这样设计：

## 15.1 第一阶段
先做短期记忆：
- `messages`
- `MemorySaver`
- `thread_id`

## 15.2 第二阶段
再加长期记忆：
- `user_profile` 存数据库
- `episodic_memory` 存向量库
- 新问题先召回再回答

## 15.3 第三阶段
再做记忆管理：
- 对话摘要
- 记忆抽取
- 去重更新
- 过期清理
