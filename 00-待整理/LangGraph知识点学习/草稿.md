以下是为第一阶段“L1：基础信息查询”定制的知识点详解，结合你们的APM场景，力求通俗易懂、案例先行。

### 0.1.1 LangGraph 基础

LangGraph 的核心思想是**用有向图来编排大模型和工具的协作流程**。图由**节点**（执行单元）和**边**（控制流转）组成，并通过**状态**（State）来记录和传递上下文。

#### 0.1.1.1 StateGraph

`StateGraph` 是整个应用的容器，你需要先定义一个**状态结构**（一个字典或数据类），然后基于它创建图。状态会沿着节点自动传递并累积更新。

**案例：定义一次查询的状态**

```python
from typing import TypedDict, Annotated
from langgraph.graph import StateGraph

# 定义状态：存储用户输入、工具查询结果、最终回复
class QueryState(TypedDict):
    question: str            # 用户原始问题
    tool_name: str           # 模型选择的工具名
    tool_args: dict          # 模型生成的工具参数
    tool_result: str         # 工具执行后返回的结果
    answer: str              # 最终生成的自然语言回答

# 创建图，声明使用该状态
graph = StateGraph(QueryState)
```

状态中的每个字段都可以被节点的返回值**更新**（默认覆盖，也可以配置为追加等）。

#### 0.1.1.2 节点

节点是图里的**执行单元**，可以是：
- **普通函数**：接收 State，返回 State 的部分更新
- **大模型调用**：利用 LangChain 的 ChatModel 生成回复
- **工具调用**：执行数据库查询等

节点用 `graph.add_node("节点名", 函数)` 添加。

**案例：添加一个“调用模型”的节点**

```python
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4o", temperature=0)

# 节点函数：调用LLM分析问题
def analyze_question(state: QueryState) -> dict:
    # 这里用简单的 prompt 示意，实际会结合工具一起使用
    response = llm.invoke(state["question"])
    return {"answer": response.content}

graph.add_node("analyze", analyze_question)
```

之后还可以添加执行工具、格式化回答等节点。

#### 0.1.1.3 条件边

条件边让流程**根据当前状态动态路由**到不同节点。例如：如果用户需要查 CPU，就去 VictoriaMetrics；如果需要查日志，就去 Loki。

添加条件边用 `graph.add_conditional_edges("源节点名", 路由函数, 映射字典)`。

**案例：根据选择的工具名分发到不同数据源**

```python
# 路由函数：从状态中取出工具名，返回字符串
def route_tool(state: QueryState) -> str:
    tool = state["tool_name"]
    if tool == "query_victoria_metrics":
        return "victoria_node"
    elif tool == "query_loki":
        return "loki_node"
    elif tool == "query_mysql":
        return "mysql_node"
    else:
        return "fallback_node"

# 添加条件边：从意图识别节点出发，分发到不同的数据源节点
graph.add_conditional_edges(
    "intent_recognition",
    route_tool,
    {
        "victoria_node": "victoria_node",
        "loki_node": "loki_node",
        "mysql_node": "mysql_node",
        "fallback_node": "fallback_node"
    }
)
```

结合之前的节点定义，最后需要设置**入口点**和**结束点**并编译：

```python
graph.set_entry_point("intent_recognition")
graph.set_finish_point("answer_formatter")
app = graph.compile()
```

### 0.1.2 工具定义

在 LangChain 中，**工具（Tool）** 就是把普通的 Python 函数封装成大模型可以“看见”并调用的格式。每个工具需要提供：
- 函数名和参数描述（给模型看）
- 实际执行的函数体

通过 `@tool` 装饰器可以快速创建。

**案例：封装一个查询交换机 CPU 利用率的工具**

```python
from langchain.tools import tool
import requests  # 假设你已封装好 VictoriaMetrics 的 HTTP 查询

@tool
def query_cpu_usage(device_ip: str, time_range: str = "5m") -> str:
    """
    查询指定交换机的CPU利用率。
    :param device_ip: 设备管理IP，如 '10.1.1.1'
    :param time_range: 时间范围，如 '5m' 表示5分钟，'1h' 表示1小时
    :return: CPU利用率百分比
    """
    # 实际实现：构造 PromQL，请求 VictoriaMetrics
    promql = f'avg(rate(switch_cpu_utilization{{instance="{device_ip}"}}[{time_range}]))'
    # ... 执行查询并返回格式化结果 ...
    return f"交换机 {device_ip} 在过去 {time_range} 的平均 CPU 利用率：23%"
```

用同样的方式封装端口丢包、设备状态等工具。**描述越清晰、参数含义越明确，模型调用的准确性就越高**。

### 0.1.3 Function Calling

Function Calling 是大模型的一项能力：当你把工具列表传给模型时，它可以**自动决定是否调用工具、调用哪个、并生成符合格式的参数**。LangChain 的 `bind_tools` 方法可以将工具绑定到模型上，让模型输出工具调用指令。

**案例：让模型根据问题选择工具并生成参数**

```python
from langchain_openai import ChatOpenAI
from langchain_core.messages import HumanMessage

# 准备好你的工具列表
tools = [query_cpu_usage, query_port_packet_loss]  # 你定义的所有工具

# 创建模型并绑定工具
llm_with_tools = ChatOpenAI(model="gpt-4o", temperature=0).bind_tools(tools)

# 模拟用户输入
user_question = "查一下核心交换机 10.1.1.1 最近1小时的 CPU 利用率"

# 调用模型，它会直接返回一个 AIMessage，其中可能包含 tool_calls
ai_msg = llm_with_tools.invoke([HumanMessage(content=user_question)])

# 检查是否有工具调用
if ai_msg.tool_calls:
    first_call = ai_msg.tool_calls[0]
    print(f"模型决定调用工具: {first_call['name']}")
    print(f"生成的参数: {first_call['args']}")
    # 输出示例：
    # 模型决定调用工具: query_cpu_usage
    # 生成的参数: {'device_ip': '10.1.1.1', 'time_range': '1h'}
```

然后你就可以用这些参数去真正执行工具函数，把结果再回传给模型，让它生成最终的自然语言回答。这正是 LangGraph 节点可以串联起来做的事情。

### 0.1.4 基础 Prompt 工程

Prompt 是告诉模型“你是谁、能做什么、应该怎么思考”的系统指令。对于第一阶段，Prompt 的核心目标是**约束模型的输出格式**，提高工具选择的准确性。

**案例：一个简洁但有效的系统 Prompt**

```python
SYSTEM_PROMPT = """
你是APM网络运维助手，专门负责查询网络设备的状态和数据。
你只能使用提供的工具来回答用户问题，绝对不要编造数据。
如果用户的问题不明确，请要求澄清；如果工具返回错误，请如实说明。

请严格遵循以下步骤：
1. 分析用户问题，提取关键参数（设备名、端口名、时间范围等）。
2. 如果参数足够，选择一个最合适的工具并生成正确的参数。
3. 如果问题涉及多个步骤，先只做第一步，等待确认。
4. 最终回答时，先给出结论，再附上原始数据摘要。

可用工具：查询CPU利用率、查询端口丢包、查询设备状态……
"""
```

将这个 System Prompt 作为消息列表的第一条消息传递给模型，可以显著降低模型的“自由发挥”概率。

### 0.1.5 简单的状态管理

在 LangGraph 中，状态（State）会随着节点的执行自动更新，并可以在后续节点中读取。你需要合理设计哪些字段要保留，哪些字段可以用完后清理，避免状态膨胀。

**案例：状态在节点间的流转**

```python
# 节点1：识别意图并写入 tool_name 和 tool_args
def intent_recognition(state: QueryState) -> dict:
    # 调用LLM分析，获取工具名和参数（实际通过function calling获得）
    # 这里模拟返回
    return {
        "tool_name": "query_cpu_usage",
        "tool_args": {"device_ip": "10.1.1.1", "time_range": "1h"}
    }

# 节点2：执行工具，读取 state["tool_name"] 和 state["tool_args"]
def execute_tool(state: QueryState) -> dict:
    tool_name = state["tool_name"]
    args = state["tool_args"]
    # 根据名称分发到实际函数
    if tool_name == "query_cpu_usage":
        result = query_cpu_usage.invoke(args)
    # ... 其他工具
    return {"tool_result": result}

# 节点3：生成回答，读取 state["tool_result"] 和 state["question"]
def format_answer(state: QueryState) -> dict:
    # 将结果和问题组合发给LLM，生成友好回复
    prompt = f"用户问：{state['question']}\n数据：{state['tool_result']}\n请组织成流畅的答复。"
    answer = llm.invoke(prompt).content
    return {"answer": answer}

# 添加到图中
graph.add_node("intent", intent_recognition)
graph.add_node("execute", execute_tool)
graph.add_node("format", format_answer)

# 简单的顺序边（第一阶段可以先不用条件边）
graph.set_entry_point("intent")
graph.add_edge("intent", "execute")
graph.add_edge("execute", "format")
graph.set_finish_point("format")
```

这样，一次查询的完整路径就走通了。后续可以在这个骨架上加入条件边和更复杂的路由，逐步向 L2 迈进。

---

**小结**
- **StateGraph** 定义了应用的骨架和共享状态。
- **节点** 承载了 LLM 调用、工具执行、格式转换等具体逻辑。
- **条件边** 负责根据状态动态调度（尽管 L1 可以先用顺序边，但它是 L2 的必备）。
- **工具定义** 和 **Function Calling** 是 AI 操作你后端数据的“手”。
- **Prompt** 是约束这双“手”不乱动的“脑”。
- **状态管理** 让多步过程有记忆，协同工作。

第一阶段完成后，你们就有了一个能听懂自然语言、精确查询单一运维指标的 AI 助手。建议先用 20 条真实查询反复测试，调优 Prompt 和工具描述，确保准确率 >95% 再进入下一阶段。

### 0.1.6 一张图解释两者的关系

```
                  ┌─────────────────────────────────┐
                  │       StateGraph (蓝图)          │
                  │   - 定义了节点、边、状态结构      │
                  │   - 应用启动时构建，编译一次      │
                  └──────────────┬──────────────────┘
                                 │ 编译后得到 app
                                 │
         ┌───────────────────────┼───────────────────────┐
         │                       │                       │
   用户A 调用               用户B 调用               用户C 调用
 app.invoke(                app.invoke(                app.invoke(
   state,                     state,                     state,
   config={"thread_id": "A"}  config={"thread_id": "B"}  config={"thread_id": "C"}
 )                         )                         )
         │                       │                       │
         ▼                       ▼                       ▼
    State 实例 A             State 实例 B             State 实例 C
  (独立的数据)              (独立的数据)             (独立的数据)
```

- **StateGraph**：定义一次，包含整个诊断流程的逻辑（节点、边、路由），无论多少用户都用同一套。
- **State**：每次调用时传入的**状态数据**。每个用户（或每个会话）有自己独立的状态，通过 `thread_id` 区分。

### 0.1.7 如何在代码中实现用户隔离

```python
# 1. 定义蓝图（只做一次）
from langgraph.graph import StateGraph

class QueryState(TypedDict):
    question: str
    tool_result: str
    answer: str

graph = StateGraph(QueryState)
# ... 添加节点和边 ...
app = graph.compile(checkpointer=SqliteSaver.from_conn_string("checkpoints.db"))

# 2. 不同用户调用时，使用不同的 thread_id
# 用户张三的查询
app.invoke(
    {"question": "核心交换机CPU利用率"},
    config={"configurable": {"thread_id": "user_zhangsan_session1"}}
)

# 用户李四的查询（完全隔离，互不影响）
app.invoke(
    {"question": "端口Gi1/0/23丢包率"},
    config={"configurable": {"thread_id": "user_lisi_session1"}}
)
```

借助 Checkpointer（如上面用的 `SqliteSaver` 或 `RedisSaver`），LangGraph 会按 `thread_id` 持久化每个会话的状态。这样即使服务重启，用户也能继续之前的对话。

### 0.1.8 为什么不每个用户建一个 StateGraph？

- **效率**：图编译涉及内部优化，每次都编译开销巨大。
- **管理**：工具、节点、路由逻辑是通用的，一份蓝图更容易维护和调试。
- **设计哲学**：LangGraph 刻意将**流程控制**（图）和**数据**（状态）分离，让你能专注于业务状态的设计。

---

**总结一句话**：StateGraph 是“生产线”，一次性建好；每个用户带着自己的“原料”（状态）走一遍生产线，最后得到自己的“成品”（回答），彼此互不干扰。





这是一个非常容易混淆的概念，因为很多文章会混着说“Agent节点”、“多Agent图”。我们用你最能理解的方式解释：

---

### 0.1.9 最简单的理解

- **节点（Node）**：是 LangGraph 图里的**一个执行步骤**，就是一段 Python 函数。
- **Agent（智能体）**：是一个**能自主思考和做事的角色**（比如“诊断网络问题的专家”）。

一个 Agent **可以由一个或多个节点组成**，只是给这些节点合作完成的任务起了一个名字。

---

### 0.1.10 用你的 APM 场景类比

想象你们团队有三个人：

| 角色（Agent） | 负责的事情 | 用 LangGraph 实现时可能需要的节点 |
|:---|:---|:---|
| **数据查询 Agent** | 听懂用户问题，去查 CPU、内存、丢包 | ① 意图识别节点（调用 LLM）<br>② 工具执行节点（调用数据库）<br>③ 回答生成节点 |
| **诊断分析 Agent** | 根据数据找异常、分析根因 | ① 异常检测节点（算法/规则）<br>② 证据关联节点（LLM）<br>③ 报告生成节点 |
| **调度 Agent（Supervisor）** | 决定把问题交给哪个 Agent | ① 任务分解节点（LLM）<br>② 路由决策节点 |

- 每个 Agent 是**逻辑上独立完成一个子任务的“专家”**。
- 每个节点是**执行专家工作的一个具体动作**。

---

### 0.1.11 三种常见的对应关系

#### 0.1.11.1 单 Agent = 单节点（最简单）
如果 Agent 做的事情极简单（比如只调用一个 API 返回结果），可以就是一个节点。

```python
# 一个节点就是一个简单的Agent
graph.add_node("simple_query_agent", call_api_node)
```

#### 0.1.11.2 单 Agent = 多节点（最常见）
一个 Agent 内部有多个步骤，这些步骤各是一个节点，但对外统称为该 Agent。

```python
# 三个节点共同组成“数据查询Agent”
graph.add_node("intent", intent_node)      # 步骤1
graph.add_node("execute", execute_node)    # 步骤2
graph.add_node("format", format_node)      # 步骤3
# 这三个节点串起来，整体就是“数据查询Agent”
```

#### 0.1.11.3 单 Agent = 一个子图（Subgraph）
把上述多个节点打包成一个“子图”，可以在主图中像普通节点一样调用，这就是 LangGraph 的**Subgraph**概念。

```python
# 把几个节点打包成一个子图，就是一个独立的Agent
diagnosis_subgraph = StateGraph(DiagnosisState)
# ... 给子图加节点、边 ...
diagnosis_agent = diagnosis_subgraph.compile()   # 这就是一个独立的Agent

# 在主图中当作一个节点使用
main_graph.add_node("diagnosis_agent", diagnosis_agent)
```

### 0.1.12 你们第一阶段的情况

在 L1 阶段，你们只需要**一个 Agent**（就是“数据查询助手”）。它内部有三个节点：

```
[意图识别节点] → [工具执行节点] → [回答生成节点]
```

这个整体你可以叫它“APM查询Agent”，但它是由 3 个节点组成的。

到了 L4 多 Agent 阶段，你会有一个主 Agent 调度多个子 Agent，每个子 Agent 又是一个子图（里面又有很多节点）。就像总公司调度多个部门，每个部门内部有多个岗位。

---

### 0.1.13 一句话总结

> **节点是函数，Agent 是角色。一个 Agent 可以由一个或多个节点（或子图）组成，节点是用来实现 Agent 能力的具体步骤。**
