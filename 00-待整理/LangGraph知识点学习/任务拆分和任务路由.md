在多 Agent 协作架构中，入口 Agent（通常称为 **Planner** 或 **Dispatcher**）负责将用户的复杂请求拆解为一组可独立执行的子任务。你问的三个问题环环相扣，下面以 LangGraph 为底座，给出一个可以直接落地的设计方案。

---

## 0.1 一、入口 Agent 的输出是什么？

入口 Agent 的输出是一个**结构化的子任务列表**（任务计划）。每个子任务包含：

- **任务类型 / 意图**：决定交给哪个子 Agent（如 `"network_diagnosis"`、`"order_query"`）。
- **任务参数 / 槽位**：子 Agent 执行所需的具体信息（设备名、订单号等）。
- **执行顺序与依赖**：可选，用于编排并行或串行。

在 LangGraph 的共享状态中，可以用一个 Pydantic 列表表示：

```python
from pydantic import BaseModel
from typing import Literal, Optional, List

class SubTask(BaseModel):
    task_id: int
    intent: Literal["network_diagnosis", "order_query", "chitchat"]  # 子Agent类型
    params: dict                     # 任务参数，如 {"device":"前端"}
    depends_on: List[int] = []       # 依赖的task_id列表（用于排序）
```

入口 Agent 的节点函数输出这个列表，并写入状态：

```python
class OrchestratorState(TypedDict):
    messages: Annotated[list[BaseMessage], add_messages]
    tasks: List[SubTask]            # 拆分后的子任务队列
    current_task_id: int
    task_results: dict              # 子任务执行结果聚合 {task_id: result}
    final_response: Optional[str]
```

**入口节点示例**（使用结构化输出）：

```python
def planner_node(state: OrchestratorState):
    user_msg = state["messages"][-1].content
    planner_prompt = ChatPromptTemplate.from_messages([
        ("system", """你是一个任务规划器。将用户请求拆分成子任务，每个子任务对应一个专业Agent。
        可用Agent类型：network_diagnosis（网络诊断）, order_query（订单查询）。
        输出JSON数组，每个元素包含 intent, params（键值对）, depends_on（可选）。"""),
        ("user", "{input}")
    ])
    chain = planner_prompt | llm.with_structured_output(...)  # 输出 List[SubTask]
    tasks = chain.invoke({"input": user_msg})
    return {
        "tasks": tasks,
        "current_task_id": 0,
        "task_results": {}
    }
```

例如输入 `"前端丢包，顺便查下订单ORD-123456"`，输出可能是：

```json
[
  {"task_id": 1, "intent": "network_diagnosis", "params": {"device": "前端", "symptom": "丢包"}},
  {"task_id": 2, "intent": "order_query", "params": {"order_id": "ORD-123456"}}
]
```

## 0.2 二、如何将任务分给不同的 Agent？

使用 **“调度循环”** 实现：从 `tasks` 队列中逐个取出任务，根据 `intent` 字段路由到对应的子 Agent 节点/子图，执行完毕后返回调度节点取下一个任务，直到队列为空。

图结构示意：

```
START → planner → dispatcher → 条件路由 → 子Agent节点/子图
                    ↑              |                |
                    └── task_complete ←──────────────┘ (执行后回到dispatcher)
                    (所有任务完成) → aggregator → END
```

### 0.2.1 调度节点（Dispatcher）

```python
def dispatcher(state: OrchestratorState):
    # 找到下一个未执行的任务（简单按顺序）
    tasks = state["tasks"]
    results = state["task_results"]
    for task in tasks:
        if task.task_id not in results:
            return {"current_task_id": task.task_id}
    return {"current_task_id": -1}  # 全部完成
```

### 0.2.2 条件路由：根据当前任务意图分发

```python
def route_to_agent(state: OrchestratorState):
    if state["current_task_id"] == -1:
        return "aggregator"
    # 找到当前任务
    current = next(t for t in state["tasks"] if t.task_id == state["current_task_id"])
    return current.intent  # 直接返回子Agent名称
```

### 0.2.3 子 Agent 执行完毕后，回到调度节点

每个子 Agent 节点最后都将自己的结果写入 `task_results`，然后无条件连回 `dispatcher`：

```python
builder.add_edge("network_diagnosis_agent", "dispatcher")
builder.add_edge("order_query_agent", "dispatcher")
```

调度节点再找下一个未完成任务，直到全部完成，跳转到聚合节点。

## 0.3 三、子 Agent 如何接收任务，输入来自哪里？

子 Agent 的输入**全部来自 LangGraph 的共享状态**，无需额外的消息队列。具体包括：

1. **当前子任务定义**：从状态 `tasks` 和 `current_task_id` 读取，包含 `intent` 和 `params`。
2. **用户原始上下文**：`messages` 中的初始用户提问（及可能的澄清历史）。
3. **前置任务的结果**：如果需要依赖，从 `task_results` 中根据 `depends_on` 提取。

在子 Agent 节点内部，**将这些信息组装成 prompt / 工具调用输入**。

### 0.3.1 子 Agent 节点实现示例（网络诊断）

```python
def network_diagnosis_node(state: OrchestratorState):
    # 1. 获取当前任务
    current = next(t for t in state["tasks"] if t.task_id == state["current_task_id"])
    params = current.params  # {"device":"前端", "symptom":"丢包"}
    # 2. 构造此子Agent的专用消息（可以是重新开始的，也可以沿用历史）
    #    这里示范：将任务参数作为新消息注入
    task_msg = HumanMessage(content=f"请诊断：设备 {params['device']} 出现 {params['symptom']}，请找出原因。")
    # 3. 调用子图（假设 network_subgraph 是之前定义好的 ReAct Agent）
    #    传入的消息可以只包含任务描述，也可以带上之前的用户原始消息
    sub_input = {"messages": [task_msg]}
    result = network_subgraph.invoke(sub_input)
    # 4. 将结果写入状态
    output = result["messages"][-1].content
    state["task_results"][current.task_id] = output
    # 5. 可选：将子Agent的输出追加到全局messages，方便最终聚合
    return {"task_results": state["task_results"],
            "messages": [AIMessage(content=f"[{current.intent}] 结果：{output}")]}
```

**子 Agent 的输入来源总结**：

| 信息 | 来源 | 如何获取 |
|------|------|----------|
| 任务意图与参数 | `state["tasks"]` + `state["current_task_id"]` | 节点内按 ID 索引 |
| 用户原始上下文 | `state["messages"]` | 可直接引用，或过滤出 HumanMessage |
| 前置任务结果 | `state["task_results"]` + `task.depends_on` | 根据依赖列表抽取拼接 |
| 系统指令/领域知识 | 硬编码或从配置读取 | 在子图 system prompt 中注入 |

## 0.4 四、完整图构建代码框架

```python
builder = StateGraph(OrchestratorState)

builder.add_node("planner", planner_node)
builder.add_node("dispatcher", dispatcher)
builder.add_node("network_diagnosis_agent", network_diagnosis_node)
builder.add_node("order_query_agent", order_query_node)
builder.add_node("aggregator", aggregator_node)

builder.add_edge(START, "planner")
builder.add_edge("planner", "dispatcher")

builder.add_conditional_edges("dispatcher", route_to_agent, {
    "network_diagnosis": "network_diagnosis_agent",
    "order_query": "order_query_agent",
    "aggregator": "aggregator"
})

builder.add_edge("network_diagnosis_agent", "dispatcher")
builder.add_edge("order_query_agent", "dispatcher")
builder.add_edge("aggregator", END)
```

## 0.5 五、进阶：并行执行与依赖管理

如果任务无相互依赖，可以在 `dispatcher` 中同时标记多个可执行任务，利用 LangGraph 的并行节点执行能力（如 `Send` API）。但最简单的是顺序执行，对多数对话场景足够。

---

## 0.6 六、总结

| 问题 | 答案 |
|------|------|
| 入口Agent输出什么？ | 结构化的子任务列表（`List[SubTask]`），每个包含 `intent` 和 `params` |
| 如何分发给不同Agent？ | 调度循环 + 条件路由，根据 `intent` 进入对应子Agent节点 |
| 子Agent输入来自哪里？ | 共享状态：当前任务描述、用户原始消息、前置任务结果 |

这套架构完全在 LangGraph 的状态管理内闭环，没有外部依赖，逻辑清晰、易于追踪，是目前多 Agent 协作的主流实现方式。


在已有多 Agent 编排框架中，要实现 Agent 之间的依赖，核心在于**调度器对 `depends_on` 的感知**和**子 Agent 对上游结果的引用**。下面基于之前定义的 `SubTask` 模型，详细说明依赖的实现方式。

---

## 0.7 一、依赖关系的定义

`SubTask` 中已有 `depends_on` 字段，它列出当前任务所依赖的 `task_id` 列表。例如：

```json
[
  {"task_id": 1, "intent": "network_diagnosis", "params": {"device": "前端", "symptom": "丢包"}, "depends_on": []},
  {"task_id": 2, "intent": "recommend_fix", "params": {}, "depends_on": [1]},
  {"task_id": 3, "intent": "order_query", "params": {"order_id": "ORD-123"}, "depends_on": []}
]
```

任务2需要等任务1完成，任务1和3无依赖，可并行（或顺序无关）。

---

## 0.8 二、调度器改造：依赖检查与跳过

原调度器只是按 `task_id` 顺序取第一个未执行的任务，现在需要**循环扫描任务列表，找到所有依赖已满足的未执行任务**。若找不到任何可执行任务且还有未完成任务，说明存在死锁或未满足的依赖，此时应报错或进入等待状态（对于纯对话系统，通常不会出现，因为 Planner 应保证无环）。

### 0.8.1 改造后的调度器逻辑

```python
def dispatcher(state: OrchestratorState):
    tasks = state["tasks"]
    results = state["task_results"]
    
    for task in tasks:
        # 跳过已完成
        if task.task_id in results:
            continue
        # 检查依赖是否全部完成
        if all(dep_id in results for dep_id in task.depends_on):
            return {"current_task_id": task.task_id}
    
    # 如果没有可执行的任务且仍有未完成，可能是依赖不满足或死锁
    unfinished = [t for t in tasks if t.task_id not in results]
    if unfinished:
        # 可记录错误或触发兜底策略，例如直接按顺序强制执行剩余任务
        # 此处为简化，强制执行第一个未完成的任务（打破依赖）
        return {"current_task_id": unfinished[0].task_id}
    
    # 全部完成
    return {"current_task_id": -1}
```

这样，任务2只有在任务1完成后才会被调度。

---

## 0.9 三、子 Agent 获取依赖任务的结果

在子 Agent 节点中，可以通过 `task.depends_on` 从 `state["task_results"]` 提取上游结果，并将其注入到当前的 prompt 或工具调用中。

```python
def network_fix_node(state: OrchestratorState):
    current = next(t for t in state["tasks"] if t.task_id == state["current_task_id"])
    
    # 获取依赖任务的结果
    dependency_context = ""
    for dep_id in current.depends_on:
        dep_result = state["task_results"].get(dep_id)
        if dep_result:
            dependency_context += f"[前置任务 {dep_id} 的结果]: {dep_result}\n"
    
    # 构造任务指令，包含历史结果
    prompt = f"根据诊断结果: {dependency_context}，请给出修复建议。"
    
    # 调用子Agent（可能是简单的 LLM 调用或子图）
    response = llm.invoke(prompt)
    
    state["task_results"][current.task_id] = response.content
    return {"task_results": state["task_results"]}
```

## 0.10 四、处理并行与依赖的混合

如果希望无依赖任务并行执行，LangGraph 原生支持通过 `Send` API 启动多个并行分支，但调度循环也可以自然地支持“伪并行”（快速顺序执行，对外部 API 可异步）。一个更强大的调度方式是：

- 在 `dispatcher` 返回一个**列表** `current_task_ids`，然后图的条件边使用 `Send` 将每个任务分发到对应的子 Agent 实例。
- 所有子 Agent 执行完毕后，通过一个 `join` 节点收集结果，再回到 `dispatcher`。

由于这涉及 LangGraph 的并行映射（map-reduce）模式，对于对话系统，顺序执行通常足够且易于调试，此处不展开并行实现。

---

## 0.11 五、避免循环依赖与死锁

Planner 节点生成任务图时应保证无环，但为确保系统鲁棒性，可以在调度器中增加保护：

- 设置最大循环次数（如 10 次），超过后强制终止并返回部分结果。
- 或检查依赖图中是否存在未执行但永远不会满足的任务（如依赖自身或依赖的任务 id 不存在），记录告警。

## 0.12 六、完整调度流程示例

假设任务列表：
- T1: 网络诊断，无依赖
- T2: 修复建议，依赖 T1
- T3: 订单查询，无依赖

**执行步骤**：
1. `planner` 生成任务列表。
2. `dispatcher` 首次扫描：T1 和 T3 无依赖，假设选中 T1（按顺序先返回），`current_task_id = 1`。
3. 路由到 `network_diagnosis_agent`，执行完毕，结果写入 `task_results[1]`。
4. 返回 `dispatcher` 再次扫描：T1 完成，T2 依赖满足（dep_id=1 在 results 中），T3 也可执行，假设选中 T2（顺序优先），`current_task_id = 2`。
5. 路由到 `recommend_fix_agent`，执行时从 `state` 中拿到 T1 的结果，生成修复建议，结果写入。
6. 回到 `dispatcher`，最后执行 T3。
7. 全部完成，`current_task_id = -1`，进入聚合。

---

## 0.13 七、总结

| 实现要素 | 方法 |
|----------|------|
| **依赖定义** | `SubTask.depends_on` 列表 |
| **调度控制** | 调度器循环检查所有未执行任务，选择依赖已满足的 |
| **数据传递** | 子Agent从 `state["task_results"]` 按依赖ID提取上游结果 |
| **异常处理** | 最大循环次数、死锁检测、兜底顺序执行 |

这种设计将任务依赖关系显式化，通过 LangGraph 的状态管理和条件路由实现了多 Agent 之间的有序协作。
