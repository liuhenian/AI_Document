# 1 意图识别
## 1.1 意图识别节点的设计
判断“第一轮解析后是否意图模糊”，通常依赖于**置信度、槽位完整性、歧义标志**三个维度的组合判定。在 LangGraph 的意图解析节点内部，你可以用如下逻辑来实现。



---

### 1.1.1 直接利用 LLM 结构化输出中的判断字段

最简单的方法：让 LLM 在结构化输出中直接给出 `needs_clarification` 和 `confidence`，解析节点无条件采纳。

```python
class IntentSlots(BaseModel):
    intent: Literal["order_query", "product_recommend", "complaint", "chitchat", "unknown"]
    confidence: float
    needs_clarification: bool
    ...

# 在解析节点中
result: IntentSlots = llm_chain.invoke(...)
return {
    "intent": result.intent,
    "confidence": result.confidence,
    "needs_clarification": result.needs_clarification
}
```

路由时直接根据 `needs_clarification` 进入澄清分支。

---

### 1.1.2 基于置信度阈值 + 槽位校验的规则兜底

LLM 判断有时过于“自信”（哪怕信息不足也强行给一个意图），因此在解析节点中增加硬性规则做二次校验会更可靠。

```python
def parse_intent(state: AgentState):
    user_msg = state["messages"][-1].content

    # ... 规则或 LLM 得到意图结果 ...
    intent = result.intent
    confidence = result.confidence
    slots = {"order_id": result.order_id, ...}

    # 规则校验
    needs_clarification = False

    # 校验1：置信度太低
    if confidence < 0.7:
        needs_clarification = True

    # 校验2：虽然置信度高，但关键槽位缺失
    if intent == "order_query" and not slots.get("order_id"):
        needs_clarification = True
    if intent == "complaint" and (not slots.get("order_id") or not slots.get("reason")):
        needs_clarification = True

    # 校验3：意图被标记为 unknown
    if intent == "unknown":
        needs_clarification = True

    return {..., "needs_clarification": needs_clarification}
```

这样即使 LLM 没意识到需要澄清，规则也会强制触发。

---

### 1.1.3 检测多意图竞争（高级）

当用户一句话包含多个意图（例如“我的订单到哪了？顺便帮我推荐个手机壳”），模型可能只输出一个意图，导致处理不完整。你可以通过以下方式检测模糊：

- 让 LLM 输出**意图列表及其片段**（如 `[{intent, span, confidence}]`），如果有多个高置信意图且无明确主次 → 判为模糊，反问用户聚焦。
- 分析 `tool_calls`（如果使用工具调用模式）：若模型在一次决策中调用了分属不同领域的工具，可以视为意图混合，进入“意图澄清/任务拆分”子图。

---

### 1.1.4 澄清节点不应仅依赖一个布尔值

建议在状态中保留 **`clarification_reason`**，便于澄清节点生成精准反问。

```python
if needs_clarification:
    if intent == "order_query" and not slots.get("order_id"):
        clarification_reason = "missing_order_id"
    elif confidence < 0.7:
        clarification_reason = "low_confidence"
    else:
        clarification_reason = "unknown_intent"
```

澄清节点据此生成：
- `missing_order_id` → “请提供您的订单号”
- `low_confidence` → “我没太明白您的意思，可以再说详细一点吗？”
- `unknown_intent` → “抱歉，我目前无法处理这个请求，请换一种方式描述”

---

### 1.1.5 总结：判断逻辑的组合策略

| 维度     | 判断条件                          | 行动         |
| ------ | ----------------------------- | ---------- |
| 置信度    | `confidence < 0.7`            | 判模糊，反问或转人工 |
| 关键槽位   | 必要参数缺失                        | 判模糊，反问补全   |
| 意图标签   | `unknown`                     | 判模糊，引导重述   |
| LLM 自觉 | `needs_clarification == True` | 直接采纳       |

```ad-important
**推荐实践**：先用 LLM 输出意图和自评标志，再在解析节点内施加槽位规则和置信度过滤，最后将 `needs_clarification` 写入状态，供路由使用。这样既利用了语义理解的灵活性，又保证了关键逻辑的确定性。
```
