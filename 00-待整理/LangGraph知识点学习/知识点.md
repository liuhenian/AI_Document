## 1.1 LangGraph基础
### 1.1.1 stateGraph
StateGraph 是整个应用的容器，你需要先定义一个**状态结构**（一个字典或数据类），然后基于它创建图。状态会沿着节点自动传递并累积更新。

**案例：定义一次查询的状态**

```python
from typing import TypedDict, Annotated
from langgraph.graph import StateGraph

# 定义状态：存储用户输入、工具查询结果、最终回复
class QueryState(TypedDict):
    question: str            # 用户原始问题
    tool_name: str           # 模型选择的工具名
    tool_args: dict          # 模型生成的工具参数
    tool_result: str         # 工具执行后返回的结果
    answer: str              # 最终生成的自然语言回答

# 创建图，声明使用该状态
graph = StateGraph(QueryState)
```
状态中的每个字段都可以被节点的返回值**更新**（默认覆盖，也可以配置为追加等）。

#### 1.1.1.1 不同用户和State的关系

```
                  ┌─────────────────────────────────┐
                  │       StateGraph (蓝图)          │
                  │   - 定义了节点、边、状态结构      │
                  │   - 应用启动时构建，编译一次      │
                  └──────────────┬──────────────────┘
                                 │ 编译后得到 app
                                 │
         ┌───────────────────────┼───────────────────────┐
         │                       │                       │
   用户A 调用               用户B 调用               用户C 调用
 app.invoke(                app.invoke(                app.invoke(
   state,                     state,                     state,
   config={"thread_id": "A"}  config={"thread_id": "B"}  config={"thread_id": "C"}
 )                         )                         )
         │                       │                       │
         ▼                       ▼                       ▼
    State 实例 A             State 实例 B             State 实例 C
  (独立的数据)              (独立的数据)             (独立的数据)
```

- **StateGraph**：定义一次，包含整个诊断流程的逻辑（节点、边、路由），无论多少用户都用同一套。
- **State**：每次调用时传入的**状态数据**。每个用户（或每个会话）有自己独立的状态，通过 `thread_id` 区分。

#### 1.1.1.2 如何在代码中实现用户隔离

```python
# 1. 定义蓝图（只做一次）
from langgraph.graph import StateGraph

class QueryState(TypedDict):
    question: str
    tool_result: str
    answer: str

graph = StateGraph(QueryState)
# ... 添加节点和边 ...
app = graph.compile(checkpointer=SqliteSaver.from_conn_string("checkpoints.db"))

# 2. 不同用户调用时，使用不同的 thread_id
# 用户张三的查询
app.invoke(
    {"question": "核心交换机CPU利用率"},
    config={"configurable": {"thread_id": "user_zhangsan_session1"}}
)

# 用户李四的查询（完全隔离，互不影响）
app.invoke(
    {"question": "端口Gi1/0/23丢包率"},
    config={"configurable": {"thread_id": "user_lisi_session1"}}
)
```

### 1.1.2 节点
节点是图里的**执行单元**，可以是：
- **普通函数**：接收 State，返回 State 的部分更新
- **大模型调用**：利用 LangChain 的 ChatModel 生成回复
- **工具调用**：执行数据库查询等

节点用 `graph.add_node("节点名", 函数)` 添加。

**案例：添加一个“调用模型”的节点**

```python
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4o", temperature=0)

# 节点函数：调用LLM分析问题
def analyze_question(state: QueryState) -> dict:
    # 这里用简单的 prompt 示意，实际会结合工具一起使用
    response = llm.invoke(state["question"])
    return {"answer": response.content}

graph.add_node("analyze", analyze_question)
```

之后还可以添加执行工具、格式化回答等节点。

#### 1.1.2.1 agent和节点之间的关系

**单 Agent = 单节点（最简单）**
如果 Agent 做的事情极简单（比如只调用一个 API 返回结果），可以就是一个节点。

```python
# 一个节点就是一个简单的Agent
graph.add_node("simple_query_agent", call_api_node)
```

**单 Agent = 多节点（最常见）**
一个 Agent 内部有多个步骤，这些步骤各是一个节点，但对外统称为该 Agent。

```python
# 三个节点共同组成“数据查询Agent”
graph.add_node("intent", intent_node)      # 步骤1
graph.add_node("execute", execute_node)    # 步骤2
graph.add_node("format", format_node)      # 步骤3
# 这三个节点串起来，整体就是“数据查询Agent”
```

**单 Agent = 一个子图（Subgraph）**
把上述多个节点打包成一个“子图”，可以在主图中像普通节点一样调用，这就是 LangGraph 的**Subgraph**概念。

```python
# 把几个节点打包成一个子图，就是一个独立的Agent
diagnosis_subgraph = StateGraph(DiagnosisState)
# ... 给子图加节点、边 ...
diagnosis_agent = diagnosis_subgraph.compile()   # 这就是一个独立的Agent

# 在主图中当作一个节点使用
main_graph.add_node("diagnosis_agent", diagnosis_agent)
```

**一句话总结**

> **节点是函数，Agent 是角色。一个 Agent 可以由一个或多个节点（或子图）组成，节点是用来实现 Agent 能力的具体步骤。**

## 2.6 第三阶段

## 2.7 阶段四
==**目标**==
开放式诊断/规划/操作意图  
• “研发中心网络很卡，帮我分析原因”  
• “画出到官网服务器的路径及延迟”

3. **路径分析需拓扑感知**：需要结合网络拓扑数据（MySQL中存储的设备连接关系）动态构建路径。  
4. **操作意图的安全与确认**：“找出未使用端口”可能涉及配置变更风险，需人工确认环节。
